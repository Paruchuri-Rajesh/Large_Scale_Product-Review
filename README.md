# FraudLens — Large-Scale Product Review Sentiment & Fraud Detection

**Group 6** — Prerana Ramesh, Rajesh Paruchuri, Ritika Mukesh Neema, Sneha Singh
*Department of Applied Data Science, San Jose State University*

A distributed batch-and-streaming pipeline for large-scale analysis of Amazon
product reviews. The system ingests the full **McAuley-Lab Amazon Reviews
2023 Cell Phones & Accessories** dataset (11 GiB raw, **20,812,945 records**),
performs feature engineering with **Apache Spark 3.5**, and trains two
scikit-learn models tracked via **MLflow**:

- a **sentiment** classifier (TF-IDF + Logistic Regression — weighted-F1 **0.841**)
- a **fraud** detector (TF-IDF + behavioral features + Logistic Regression — **ROC-AUC 0.845**)

Real-time scoring is delivered through **Apache Kafka** + **Spark Structured
Streaming**. The system exposes **five serving surfaces**: a FastAPI REST
service with an LLM-generated fraud explanation, a Streamlit demo dashboard,
an **MCP server** for Claude Desktop / Cowork integration, an Ollama-driven
**autonomous review auditor**, and a Kafka-integrated prediction fan-out.

The full pipeline runs end-to-end in **~6 hours on a standard 16 GB laptop**.
Full IEEE-formatted build report: [`IEEE_Formatted_Report.pdf`](IEEE_Formatted_Report.pdf).

| Full real-data run | 16,413,084 | 4,105,036 | 0.680 | **0.841** | **0.845** (no leakage) | **0.356** (best) |

Post-hoc isotonic calibration further drops Brier 0.143 → **0.029** and ECE
0.302 → **0.000146** with zero ROC-AUC change (see Table IV in the report).

On top of the core loop, the repo includes:

- **ML evaluation and reporting** — baseline model comparison, fraud threshold
  tuning (`models/thresholds.json`), error-analysis exports, drift monitoring,
  fraud probability calibration (raw vs sigmoid vs isotonic), and a leakage
  ablation study — all writing to `reports/ml/`.
- **Streamlit demo dashboard** (`make streamlit`) — a single-file UI that
  loads the joblib pipelines and parquet aggregates directly, no API server
  needed.
- **MCP server** (`make mcp`) — exposes `predict_review`,
  `get_fraud_reviewers`, `get_top_products` as natural-language-callable
  tools through the Model Context Protocol. Works with **Claude Desktop**
  and with **Cowork** (task-oriented shell for non-developer users).
- **LLM review auditor** (`make audit PRODUCT=<asin>`) — autonomous agent
  using **Ollama Llama 3.2** with tool-calling. Loops up to 8 iterations,
  calling `get_product_aggregate`, `get_product_reviews`,
  `get_top_fraud_reviewers`, and `score_review` until it produces a
  structured risk verdict (HIGH / MEDIUM / LOW).
- **LLM `fraud_explanation`** field on `POST /predict` — plain-language
  summary, risk band, and bullet reasons, generated by Ollama Llama 3.2
  with a rule-based fallback if the LLM is unreachable.
- **Form → Kafka fan-out** (`PUBLISH_PREDICT_TO_KAFKA=1`) — when set, every
  `/predict` call also publishes the review to the `reviews` Kafka topic,
  so form submissions flow through Spark Structured Streaming and appear in
  `/stream/recent` in addition to returning the synchronous result.
- **Live-observability helpers** — `scripts/kafka_tap.py` (live message
  tap), `scripts/publish_one_review.py` (one-shot Kafka producer),
  `scripts/spark_bigdata_demo.py` (4-stage Spark aggregation over 20.5 M
  rows for showcasing distributed processing).

### Optional AWS cloud path

In addition to the local file-based pipeline, the project supports an
AWS-hosted alternative for the ingest + ETL stages:

- **Amazon S3 bucket** `cellphonesandaccessories` — holds the 8.7 GB
  source `.jsonl` files and the converted Parquet chunks (~45-48 MB each).
- **AWS Glue ETL job** `json-to-parquet` — converts raw JSONL → columnar
  Parquet (4 DPUs, ~2 minutes total).
- **Amazon Athena** — runs SQL directly against the S3 Parquet
  (`reviews_db.cell_phones_reviews`) for validation and exploration
  (e.g. `SELECT COUNT(*) FROM reviews_db.cell_phones_reviews;` returns
  `20,812,945`).

The downstream Spark, training, streaming, and serving stages are the
same regardless of whether the Parquet lives on local disk or S3.

## Using a real Amazon Reviews dataset

The original AWS Open Data PDS bucket (`s3://amazon-reviews-pds/`) was
**deprecated by Amazon in late 2023**. The de-facto successor is the
**McAuley-Lab Amazon Reviews 2023** corpus on Hugging Face.

### One-time data prep

```bash
# 1. Download a category file (here: Cell_Phones_and_Accessories, 8.7 GB)
curl -L -o /path/to/Cell_Phones_and_Accessories.jsonl \
  "https://huggingface.co/datasets/McAuley-Lab/Amazon-Reviews-2023/resolve/main/raw/review_categories/Cell_Phones_and_Accessories.jsonl"

# 2. Convert source schema (rating/title/text/asin/...) to project schema
#    (review_id/product_id/reviewer_id/star_rating/...). Streams line-by-line
#    so the 8.7 GB source never enters RAM. ~104k rec/sec, 0 skipped rows.
python3 -m src.ingest.import_amazon_real \
    --src /path/to/Cell_Phones_and_Accessories.jsonl \
    --out data/raw/reviews.jsonl
```

### Run the pipeline at full scale

```bash
SPARK_DRIVER_MEM=8g make etl     # ~24 min on full 20.5M cleaned rows
make train                       # ~5 hours on a 16 GB laptop
make serve                       # dashboard + REST API
```

### Spark tuning

`src/common/spark.py` ships sized for full-scale runs: **8 GB driver
memory** (override with `SPARK_DRIVER_MEM=...`), **200 shuffle partitions**,
**adaptive query execution + skewJoin** enabled, **Kryo serializer**. The
window functions in batch ETL (over `reviewer_id`, `product_id`) need this
to survive 20 M rows on a single node. For a real cluster, override via
`spark-submit --master ...` and tune partitions further.

## Models and evaluation

**Training (production path)** -- `make train` / `src/train/train.py`:

- **Sentiment** -- TF-IDF (uni + bi-gram, `min_df=10`, `max_features=50,000`)
  + multinomial logistic regression with `solver='saga'` (the recommended
  solver for large sparse problems). Labels from star rating: 1-2 negative,
  3 neutral, 4-5 positive.
- **Fraud** -- TF-IDF (`min_df=10`, `max_features=15,000`) on cleaned review
  text plus **12 numeric behavioural features** (exact column names recorded
  in `models/meta.json -> numeric_fraud_features` after a successful train;
  see also `src/train/features.py:FRAUD_MODEL_NUMERIC_FEATURES`). The
  training flow evaluates **two fraud candidates** with the same feature
  pipeline:
  - `SGDClassifier(loss='log_loss', class_weight='balanced')`
  - `LogisticRegression(solver='saga', class_weight='balanced')`

  Both are sparse-friendly and parallel — necessary for the full 16 M-row
  corpus. The original GradientBoostingClassifier and heavy
  RandomForestClassifier candidates do not finish in laptop time at this
  scale. The selection rule is unchanged: train both, save the higher F1.
  Current selected fraud model in this repo's latest trained artifacts:
  **`logreg+tfidf+behavior`**.

  Weak fraud labels come from batch heuristics in ETL plus a small
  controlled noise step (3.4% rule-positive flips, 0.15% rule-negative
  flips) to defeat memorization of the rule. The API returns
  **probabilities**; the threshold-tuning study finds **0.80** as the best
  F1 operating point on the held-out 4.1 M rows (default 0.5 is a poor
  threshold given the 3.57% class imbalance).

Both runs log to MLflow under experiment `amazon_reviews_sentiment_fraud`.

**Extra evaluation scripts** (run after ETL has produced train/test Parquet — same
schema as production):

| Script | Role |
|--------|------|
| `python -m src.train.baselines` | Compare simpler/alternate models; writes `reports/ml/baseline_comparison.csv`. |

## LLM features and agent surfaces

The repo includes three LLM-backed surfaces, all wired against a local
**Ollama** server (default `http://localhost:11434`). Switching to a hosted
LLM is a one-line change at the call site.

### `fraud_explanation` on `POST /predict`

Every single-review prediction returns an optional `fraud_explanation`
object: a plain-language summary, a `risk_level` band, and a
`feature_signals` dictionary of the inputs the LLM used. If Ollama is
unreachable, a deterministic rule-based fallback fills the same field so
the API contract never breaks.

### MCP server (`make mcp`)

`src/serve/mcp_server.py` exposes the same scoring + aggregate access as
**Claude-callable tools** through the Model Context Protocol over stdio.
Three tools are registered:

- `predict_review(...)` -- the same scorer the REST `/predict` uses.
- `get_fraud_reviewers(limit)` -- top suspicious reviewers from the ETL
  aggregates.
- `get_top_products(limit, sort_by)` -- top products, sortable by
  `review_count`, `fraud_rate`, or `product_avg_rating`.

To wire it into Claude Desktop, copy the snippet at the top of
`src/serve/mcp_server.py` into
`~/Library/Application Support/Claude/claude_desktop_config.json`.

### LLM review auditor (`make audit PRODUCT=<asin>`)

`src/agents/review_auditor.py` is an autonomous review auditor: it uses
**Ollama tool-calling** to call back into the FastAPI service for tools
(`get_product_aggregate`, `get_product_reviews`,
`get_top_fraud_reviewers`, `score_review`) and produces a structured audit
report. Live example:

## Synthetic data generation

The repo's primary corpus is the real McAuley-Lab Amazon Reviews 2023 data
(see [Using a real Amazon Reviews dataset](#using-a-real-amazon-reviews-dataset)).
The synthetic generator below is kept as a development convenience -- it
produces a self-contained JSONL so a smoke run can finish in minutes
without any download.

The generator (`src/ingest/generate_sample.py`)
produces Amazon-shaped **JSONL** locally so the full pipeline runs without a multi-GB download.
Phrase pools are sentiment-stratified; a configurable share of rows simulate fraud-like
patterns (e.g. duplicate bursts, velocity effects, reviewer rings, hard negatives). Large
runs: e.g. `--rows 1000000`; Spark scales with data size.

**Difficulty (`easy` \| `medium` \| `hard`):**

- **`medium` is the default** (`make ingest`, pipeline scripts unless overridden).
- **`easy`** — cleaner separation between classes; closer to a simplified legacy mix.
- **`medium`** — cross-class phrase overlap, typos/punctuation noise, mixed-sentiment wording,
  richer fraud mixture.
- **`hard`** — stronger overlap and subtler fraud-like patterns so models face a tougher task.

Harder modes increase realism for **sentiment confusion** and **fraud subtlety** while keeping
the **JSONL schema and downstream column layout unchanged**.

Examples:

```bash
python -m src.ingest.generate_sample --rows 30000 --fraud-share 0.06 --difficulty hard
make ingest   # uses medium by default
ROWS=50000 DIFFICULTY=easy bash scripts/run_pipeline.sh
```

## Results and interpretation

Metrics on **synthetic** data are for development and apples-to-apples comparisons—not a claim
of production accuracy. With **`medium`/`hard`** difficulty, holdout curves are **more
informative**: fraud threshold sweeps show clearer precision/recall tradeoffs than an
everywhere-easy generator, and sentiment/fraud error analysis becomes more meaningful.

The **dashboard + drift + explanation** pieces are meant for monitoring and transparency:
they do not replace rigorous evaluation on real Amazon data, but they make the current run
easier to interpret than headline numbers alone.

### Threshold sweep — fraud classifier

| Threshold | Precision | Recall | F1 |
|---|---|---|---|
| 0.50 | 0.147 | 0.721 | 0.244 |
| 0.60 | 0.193 | 0.635 | 0.296 |
| 0.70 | 0.242 | 0.549 | 0.336 |
| **0.80** (selected) | **0.292** | 0.457 | **0.356** |
| 0.90 | 0.358 | 0.265 | 0.305 |

The default `0.5` threshold over-fires (precision 14.7%). Tuning to **0.80**
trades a moderate recall hit for a 2× precision gain and the best F1 in the
sweep. The selected threshold is persisted to
[`models/thresholds.json`](models/thresholds.json).

### Probability calibration

Raw fraud probabilities are reasonable rankers but poorly calibrated. Post-hoc
isotonic calibration cuts Brier 5× and ECE by three orders of magnitude with
zero ROC-AUC change:

| Method | Brier | ECE | ROC-AUC |
|---|---|---|---|
| Raw | 0.143 | 0.302 | 0.846 |
| Sigmoid (Platt) | 0.030 | 0.002 | 0.846 |
| **Isotonic** (best) | **0.029** | **0.000146** | 0.846 |

Run `python -m src.train.calibration_report` to regenerate these on the
current holdout. Results are written to `reports/ml/fraud_calibration_*`.

## Data schema (logical view)

The pipeline's data flow is modeled as 8 logical tables spanning the
ingest-to-scoring lifecycle. Physically, they're stored as Parquet,
JSONL, joblib, and JSON files (no RDBMS required), but the logical
schema makes the dataflow easier to reason about:

| Table | Backing file(s) | Role |
|---|---|---|
| `RAW_REVIEW` | `data/raw/reviews.jsonl` | every ingested record, keyed by `review_id` |
| `FEATURED_REVIEW` | `data/processed/{train,test}.parquet` | ETL output: derived text + behavioral features + weak `fraud_label` + `sentiment_label` |
| `PRODUCT_AGG` | `data/processed/product_agg.parquet` | rolled-up per `product_id`: `review_count`, `avg_rating`, `pct_5star`, `fraud_rate` — powers `get_top_products` |
| `REVIEWER_AGG` | `data/processed/reviewer_agg.parquet` | rolled-up per `reviewer_id`: review count, avg rating, fraud rate, verified share — powers `get_fraud_reviewers` |
| `MLFLOW_RUN` | `mlruns/` | each training run: params, metrics, artifact paths |
| `MODEL_ARTIFACT` | `models/*.joblib` | serialized sklearn pipelines (sentiment + fraud) |
| `META_JSON`, `THRESHOLDS_JSON` | `models/meta.json`, `models/thresholds.json` | feature lists, selected fraud threshold (`best_f1_threshold=0.80`), calibration metadata |
| `SCORED_REVIEW` | `data/streaming_out/scored/*.json` | scored output from Spark Streaming: `fraud_proba`, `fraud_flag`, `sentiment`, `batch_id` |
| `DIAGNOSTIC_REPORT` | `reports/ml/*.{csv,json}` | drift, threshold sweep, calibration, error samples |

See Fig. 5 of the IEEE report for the full ER diagram with primary/foreign keys.

## Limitations

Honest caveats from the IEEE report (Section XI.A):

- **Single-category corpus.** Only Cell Phones & Accessories was used.
  Category-level features add little signal in this setup; a multi-category
  corpus would let category features carry real information.
- **No unhelpful-votes column.** The McAuley-Lab dump exposes only
  `helpful_votes`. We set `total_votes = helpful_votes`, which makes any
  helpfulness-ratio feature degenerate, so we don't use one.
- **Weak fraud labels.** Ground truth labels are not available, so the
  fraud target comes from rule-based heuristics. Reported metrics are
  against that proxy and should not be quoted as production fraud accuracy.
- **Sentiment LogReg did not fully converge.** The `saga` solver hit
  `max_iter` on 16 M sparse rows. The model still performs well, but
  weights may not be fully optimal.
- **No GBT.** Gradient-Boosted Trees were the original choice but did not
  finish in laptop time on 16 M rows. We use linear models, which scale
  but may miss complex non-linear patterns.

## Where this would go next

From IEEE report Section XI.B:

- **Wire the tuned threshold into serving.** `models/thresholds.json` holds
  `best_f1_threshold = 0.80`; FastAPI currently uses the default 0.5.
- **Wire calibration into serving.** Emit isotonic-calibrated `fraud_proba`
  from `/predict` so downstream consumers can trust the probability values
  for thresholding decisions.
- **Multi-category training.** Concatenate 5-10 category JSONLs so category
  features stop being constant.
- **Hand-labeled fraud sample.** Build a small (~1,000-row) human-labeled
  validation set to ground the AUC against real labels instead of a noisy
  weak-label proxy.
- **MLflow Model Registry.** Stage → production promotion, FastAPI loads
  by stage rather than by file path.
- **Docker Compose** for Spark + MLflow + Kafka + FastAPI + Ollama so the
  project deploys on any host with a single command.
- **Feature store** with real reviewer / product history at serving time
  instead of neutral defaults — closes the train/serve skew gap on the
  behavioral features.
